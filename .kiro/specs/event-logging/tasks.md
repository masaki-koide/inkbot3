# Implementation Plan

- [x] 1. CommandLog データモデルの作成
- [x] 1.1 Prisma スキーマに CommandLog モデルを追加し、マイグレーションを実行する
  - コマンド名、ギルドID・名、チャンネルID・名、ユーザーID・名、オプション、記録日時のカラムを定義する
  - コマンド名は汎用的な文字列カラムとし、今後のコマンド追加時にスキーマ変更が不要な設計にする
  - オプションカラムのみ nullable とする
  - マイグレーションを作成・適用し、Prisma クライアントを再生成する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 2.1, 2.2, 3.1, 4.1_

- [x] 2. CommandLog リポジトリの作成
- [x] 2.1 コマンドログをデータベースに保存するリポジトリを作成する
  - 既存のファクトリ関数パターンに従い、PrismaClient を受け取るファクトリ関数を定義する
  - ログ記録メソッドは INSERT のみを行い、戻り値は void とする
  - 型は ReturnType による推論で導出する
  - _Requirements: 1.1_

- [x] 2.2 (P) リポジトリのユニットテストを作成する
  - 全フィールドが正しく保存されることを検証する
  - オプションが null の場合に正常に動作することを検証する
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.2, 3.1, 4.1_

- [x] 3. コマンドハンドラへのログ記録の組み込み
- [x] 3.1 コマンドハンドラがコマンドログリポジトリを受け取れるように変更する
  - セットアップ関数のパラメータにコマンドログリポジトリを追加する
  - エントリーポイントでコマンドログリポジトリを生成し、コマンドハンドラに注入する
  - _Requirements: 1.1_

- [x] 3.2 各コマンドハンドラの先頭でイベントログを記録する
  - Interaction からギルドID・名、チャンネルID・名、ユーザーID・名、コマンド名、オプションを取得する
  - オプションは `name=value` 形式の文字列に変換する（オプションがない場合は null）
  - ログ記録処理を try-catch で囲み、失敗時はコマンド本体の処理を継続する
  - 失敗時は console.error でエラー内容を出力する
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.2, 2.3, 3.1, 4.1, 5.1, 5.2_

- [x] 3.3 コマンドハンドラの統合テストを作成する
  - コマンド実行時にログ記録が呼び出されることを検証する
  - ログ記録が失敗してもコマンド処理が正常に完了することを検証する
  - _Requirements: 5.1, 5.2_
